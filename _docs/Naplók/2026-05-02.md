- Az Appwrite `databases.createCollection(databaseId, id, name, permissions=[], documentSecurity=true, enabled=true)` API üres `permissions` + `documentSecurity: true` flaggel hozza létre a collection-t — ez a kívánatos minta a tenant-scope ACL-hez. ## Harmadik szekció — A.3 server-side implementáció

A délutáni session-ben az **A.3 (permissionSets réteg)** szerver-oldali része lement. Az ADR 0008 B blokk teljes szerver-oldal kódja kész — A.3.1–A.3.5 + A.3.7 server-side cache. ### Approach + Codex előzetes review

Az implementáció előtt 6 kérdéses kerekasztalt vittem Codex-szel ([codex:rescue review]). Megerősítette a tervet:

1. **A.3.2 best-effort failover** — log + skip rollback (mint a workflow seed). A `permissionSetsSeeded` jelzés a response-ban explicit (Codex flag).
2. **archive_permission_set: (b) opció** — junction docs maradnak, archived set-eket a `userHasPermission()` skip-eli (`archivedAt === null` szűrő). Reverzibilis, engedmény-logikájú.
3. **Per-office snapshot Map** OK — a `Map<editorialOfficeId, snapshot>` cache-kulcs jó.
4. **lookupOrgRole/lookupOrgIdFromOffice** a `permissions.js`-be (server-only export, ne duplikáljam main.js-ben).
5. **Soros early-exit lookup** (admin label → org role → permission set) — admin-heavy pathon a felesleges `Promise.all` plusz I/O.
6. **Iteratív rollout** — A.3.6 retrofit külön commit, mert sok action visselkedését egyszerre változtatja.

**Kockázati flag-ek** (mind kezelve):
- A.3.2 nem CSAK `bootstrap_organization`-ben kell, hanem `create_editorial_office`-ben is (különben aszimmetria).
- A.3.3 új `permissionSet.create/edit/archive/assign` slug-ok dead permission-ként maradnak addig, amíg a guard CF-ek nem hivatkoznak rájuk (A.3.6 hatáskör).

### A.3.5 — Két `permissions.js` modul

Mivel a `maestro-shared` ESM, a CF (CommonJS) nem tudja közvetlenül importálni — két forrás, inline duplikációval (A.7.1 phase 2: AST-equality CI test):

- **[`packages/maestro-server/.../src/permissions.js`](packages/maestro-server/functions/invite-to-organization/src/permissions.js)** (CommonJS): inline slug-set + `DEFAULT_PERMISSION_SETS` + `validatePermissionSetSlugs` duplikáció, plus async lookup helperek. Drift-hint kommentelve.

#### Async lookup helperek (server-only)

- **`buildPermissionSnapshot(databases, env, user, officeId, orgRoleByOrg?)`** — egyetlen lookup-build. 1) office → orgId, 2) user org-role, 3a) **owner/admin shortcut**: 33 slug halmaz (`OFFICE_SCOPE_PERMISSION_SLUG_SET` clone), 3b) member-path: `groupMemberships` (limit 100) → `groupPermissionSets` (lapozott, max 1000) → `permissionSets` (lapozott, `archivedAt === null` szűrt). Defense-in-depth: csak office-scope slug kerülhet a Set-be (`org.*` direct DB write ellen).
- **`userHasPermission(databases, env, user, slug, officeId, snapshotsByOffice?, orgRoleByOrg?)`** — assertOfficeScope → admin label → snapshot Set check.
- **`userHasOrgPermission(databases, env, user, slug, orgId, orgRoleByOrg?)`** — assertOrgScope → admin label → org role (owner / admin minus excluded set).
- **`createPermissionContext()`** — `{ snapshotsByOffice: Map, orgRoleByOrg: Map }` per-request scaffold.

### A.3.3 — Permission set CRUD

### Új ismeret / Mi volt nehéz

- **Két forrás, manuális szinkron**: a `DEFAULT_PERMISSION_SETS` és slug-set inline duplikációja (CF CommonJS vs shared ESM) ugyanaz a drift-rizikó, mint a `validateCompiledSlugsInline` az A.2.1-ben. A megoldás ugyanaz: explicit drift hint kommentben + Phase 2 single-source bundle (A.7.1).
- **`permissionSets.permissions[]` defense-in-depth**: a `validatePermissionSetSlugs()` a write-pathon kihagy minden `org.*` slug-ot, de a `buildPermissionSnapshot` is csak office-scope slug-ot vesz a Set-be (DevTools / direkt DB write ellen).
- **`Query.isNull('archivedAt')`** — Appwrite-on a nullable datetime mezőt így szűrjük, ezzel a `permissionSets` archived state-jét a snapshot build figyelmen kívül hagyja. Restore esetén a doc archivedAt=null lesz, így a következő snapshot-build már bevonja.

### Holnap

- A.3.6 — meglévő CF guardok lecserélése `userHasPermission()` / `userHasOrgPermission()`-re. Külön commit, magas regressziós kockázat, action-onkénti review.
- Manuális smoke teszt az A.3 csomagra: `bootstrap_permission_sets_schema` → új org létrehozása → `permissionSetsSeeded: 3` → `create_permission_set` (custom set) → `assign_permission_set_to_group` → `unassign` → `archive_permission_set` → `restore`.

### Harden pass (este → éjszaka)

A frissen implementált A.3 kódra `/harden` skill — Codex baseline + adversarial + verifikáló review-pass + manuális simplify (3 párhuzamos agent quota miatt Claude saját olvasattal).

**Javított (4):**
1. **🔴 Critical** — [permissions.js](packages/maestro-server/functions/invite-to-organization/src/permissions.js) per-request cache-kulcs (`getOrgRole`, `userHasPermission`) most `${userId}::${scopeId}` formátumú. Multi-user CF flow-ban (jövőbeli target-user validáció) az előző csak-`scopeId` kulcs auth leak-et okozhatott volna.
2. **🟠 P1** — Cursor pagination `select()`-jeiben minden lapozott query explicit `$id`-t kap (`groupMemberships`, `groupPermissionSets`, `permissionSets`, `getOrgRole`). A `cursorAfter()` az utolsó doc `$id`-jét várja; szelektív `select()` defense-in-depth-tel kerül be.
3. **🟡 P2** — `getOrgRole` lookup hibát NEM cache-eli többé; egy átmeneti DB hiba ne fagyassza le a teljes request engedélyezését. A sikeres cache-elés a `try` blokk végén; `catch` csak `return null`.
4. **⚪ Should** — `assign_permission_set_to_group` 409 fallback `null` race-en új response-action `assignment_state_unknown` (a `already_assigned` mostmár csak akkor jön, ha a verifikáló lookup ténylegesen talál junction doc-ot).

**Simplify pass — NO CHANGES** (a kód már elég tiszta). Auth-blokk duplikáció (5 új CRUD action × 12 sor) projekt-szintű minta — az egész CF-en uniform módon alkalmazott; refaktor külön scope (A.3.6 vagy general cleanup). Komment-volume (~30%) magyar komment + ADR-tracelink projekt-konvenció.

**Elutasított (NOISE):**
- Cross-tenant existence oracle (`getDocument` 404 vs 403): projekt-szintű minta (`create_group` ugyanígy), 36-byte UUID brute-force-zhatatlan. A.3.6 / general refactor scope.
- `buildOfficeAclPerms()` formatter assumption: docstring már most kimondja, current callers safe.

**Nyitott DESIGN QUESTION (3, user-döntésre vár):**
1. **Schema-evolution drift**: új slug nem terjed automatikusan a meglévő `owner_base` set-ekhez. A.7 phase 2: version + backfill, vagy single canonical source.
2. **Bootstrap partial failure no retry**: `permissionSetSeedErrors` UI-re hárítja a manual retry-t. A.4 hatáskör.
3. **`buildPermissionSnapshot` worst-case fan-out** (1000 group × 100 chunks × 10 pages): reális usage 5-10 csoport, pathologikus küszöb felett rendszer-szintű probléma. Phase 2 telemetria + hard cap.

**Verifikáció**: Codex final review — `ready to ship`. Nincs regresszió, ADR 0008 konzisztencia OK.
