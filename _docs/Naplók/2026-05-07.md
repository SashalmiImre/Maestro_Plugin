# 2026-05-07

## Mit csináltam

**Egyetlen session, sok kis észrevétel — Codex-szel szoros együttműködésben** ([[Csomagok/dashboard-workflow-designer|Dashboard]] + `invite-to-organization` CF, ~12 fájl, 1 új [[0009-membership-user-identity-denormalization|ADR]], 1 új [[Komponensek/UserIdentityMap|komponens-jegyzet]]).

### Kiadvány-create / aktivátál bug-fix lánc (séma drift)

A felhasználó új kiadványt hozott létre → **500 hibák egymás után**, root cause minden esetben **séma drift** (a CF kód többet feltételez, mint amit a Console-szerkesztett séma tartalmaz):

1. **`publications.modifiedByClientId` (string, 36, nullable) hiánya** — MCP-vel hozzáadtam élőben + a `bootstrap_publication_schema` action-be is felvettem idempotensen ([[Komponensek/PermissionHelpers]] adatmezőhöz nem köthető, csak self-echo szűréshez használja a `validate-publication-update` CF a SERVER_GUARD pattern-ben).
2. **`groups` collection 4 mező hiánya** (`color`, `isContributorGroup`, `isLeaderGroup`, `archivedAt`) — `seedGroupsFromWorkflow` `Query.select(...)` listájában szerepelnek, az Appwrite "Invalid query: Attribute not found in schema: color"-rel utasította vissza. Mind a 4 mezőt MCP-vel hozzáadtam, plus a fallback regex-et hardenel-tem a `groupSeed.js`-ben (`unknown attribute|unknown column|attribute not found|invalid query` — több hibaüzenet-variánst fog jövőben).
3. **`DEADLINES_COLLECTION_ID` env var hiánya** a CF-en — az `activate_publication` action `misconfigured` 500-mat adott. MCP `functions_create_variable`-rel pótoltam (`= deadlines`).

### `Cannot destructure property 'holidays' of 'undefined'` (folyamatos uncaught error)

A Dashboard `useUrgency` hook **csak 3 paraméterrel** hívta a `calculateUrgencyRatio`-t, miközben az 4-paraméteres (`article, deadlines, workflow, options`). Az options helyére a `workflow` csúszott; az 4. paraméter `undefined` lett, és a destructure minden cikkre, minden 5 perces tickre hibázott. Fix: `workflow` és `publication` paraméter felvétele (utóbbi az `excludeWeekends` mezőhöz — Codex stop-time review #2 hozta elő, hogy nem hardcode-olható `true`-ra a publikáció saját beállítása helyett). [[Komponensek/DataContext|DataContext.workflow]] singular state + `activePublication` lookup.

### Új CF action `change_organization_member_role` ([[0009-membership-user-identity-denormalization|ADR 0009]] B blokk)

A `Felhasználók` fülön nem volt sem CF action, sem UI a meglévő tag role-jának változtatásához (`member ↔ admin ↔ owner`). 8 védelmi rétegű új action: payload validation, **self-edit guard**, slug check (`org.member.role.change`), **owner-touch guard** (admin nem nyúl owner-hez), membership lookup, idempotens no-op, **last-owner guard**, write. UI-ban role-dropdown a tagok mellett (csak owner-callerek látják, nem self).

**Codex stop-time follow-up**: a sikeres role-change után a `members` lista stale maradt a same-tab UX-ben, mert a `useTenantRealtimeRefresh` `ORG_CHANNELS` listája nem tartalmazta az `ORGANIZATION_MEMBERSHIPS` csatornát. Codex C-kombinált megoldása: A (Realtime cross-tab) + B (explicit `loadData()` callback `handleRoleChange` finally-jében) — a `GroupsRoute.jsx` precedensét követve.

### Server-side denormalizáció ([[0009-membership-user-identity-denormalization|ADR 0009]] A blokk)

A `EditorialOfficeGroupsTab` és `OrganizationSettingsModal` `userNameMap`-jét eddig csak a `groupMemberships` táplálta (csak ott volt denormalizált `userName`/`userEmail`). Új org bootstrap után a saját user-ünk is csak `userId`-vel jelent meg a UI-on, mert még nem volt egyetlen csoportban sem.

**Csinált:**
- 4 új oszlop élőben (`organizationMemberships` + `editorialOfficeMemberships` × `userName` + `userEmail`)
- `helpers/util.js` `fetchUserIdentity(usersApi, userId, cache, log)` helper failure-tolerant + per-request cache-szel
- `main.js` `userIdentityCache` Map a `ctx`-ben
- 3 CF write-flow frissítve (snapshot-at-join): `bootstrapOrCreateOrganization`, `acceptInvite` (a meglévő `callerUserDoc`-ból direkt), `createEditorialOffice`
- Új CF action `backfill_membership_user_names` (owner-anywhere, idempotens, paginated, dryRun-támogatás)
- Frontend `userIdentity.js` helper több source-támogatáshoz + idempotens merge
- A meglévő 3 rekord backfill-jét MCP-vel manuálisan végeztem (a CF-action `x-appwrite-user-id` headert vár, ami MCP-ből biztonsági okból blokkolt)

## Mi volt nehéz / blokkoló

- **Az `appwrite-cli` tax-yes (Y/n prompt)** a `appwrite push function`-on automatizációhoz — `yes |` pipe-pal megoldva.
- **A `backfill_*` action MCP-ből nem futtatható** (`x-appwrite-user-id` header injection blokkolt). A 3 rekordot MCP `tables_db_update_row` direkt-update-tel pótoltam. A `change_organization_member_role` jövőbeni futtatáshoz egy bejelentkezett owner-rel a Dashboardról vagy az Appwrite Console "Execute function" gombjáról kell.
- **Codex worktree drift** — Codex-szel kapcsolt session-ben az `--cd` flag nélkül a `claude/...` worktree branchben futott, ami a feature-branch fájlokat NEM tartalmazta. Nyilvánvaló volt a sok "fájl nincs"-üzenetből; kommunikáltam a usernek és a fő repón folytattam.

## Új ismeret / felfedezés

- **Schema drift mintázat**: a `bootstrap_*_schema` action-ök NEM minden szükséges mezőt fednek le (a Console-ban manuálisan létrehozott séma + a kódban használt mezők szétcsúszhatnak). A `bootstrap_publication_schema`-ba vissza-pótlás kötelező minden új CF write-mező esetén.
- **Realtime fogadási minta**: a `useTenantRealtimeRefresh` `ORG_CHANNELS` lista szándékosan szűk; minden új saját mutáció után meg kell vizsgálni, hogy az érintett collection ott van-e. Plus a kódbázis preferált mintája Codex szerint: **explicit reload saját mutáció után + Realtime cross-tab szinkronra** ([[Komponensek/RealtimeBus]] + [[Csomagok/dashboard-workflow-designer]] precedens: `GroupsRoute`).
- **AppwriteException error-szöveg**: a Query.select pre-validáció `Invalid query: Attribute not found in schema: <key>` formában dob, NEM `Unknown attribute: <key>`. A schema-fallback regex-eket szélesebbre kell venni.
